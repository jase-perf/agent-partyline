# Dashboard Client Audit — 2026-04-20

Auditor: Claude (Opus 4.7, 1M ctx)
Scope: `/home/claude/projects/claude-party-line/dashboard/*.js`, `dashboard.css`,
`index.html`, `notifications.js`, `quota.ts`.
Mode: read-only; no source modified.

## Summary

`dashboard.js` is a 2,216-line single-file module with a dozen globals, five
view/render functions that all read those globals, and no dispatcher layer
between the WebSocket and rendering. The result is correct in the _steady
state_ (everything already loaded, user clicked their way in) but fragile on
every _transition_:

- **Direct-URL load to `/session/<name>` is broken by design.** `applyRoute`
  runs at module evaluation (`dashboard.js:1928`), before `connect()` at
  `:2216` has even opened the WebSocket. `lastSessions` is `[]`, so `known`
  is false and the view paints "Session '<name>' is not known to the
  dashboard." Nothing re-runs that check when the `sessions` payload later
  arrives.
- **Realtime updates are inconsistent because they take two fundamentally
  different paths.** Multicast presence flows through `updateSessions →
updateOverviewGrid` (in-place DOM diff). DB aggregator events flow through
  `session-update` which triggers a full `/api/session` refetch AND a
  `renderStream({incremental})` refetch per event — but only when viewing
  that session. Some state is keyed by `session.name`, some by
  `session.session_id`; the two never converge.
- **`updateSessions(lastSessions)` is called six times from unrelated code
  paths**, each rebuilding every card's inner DOM and re-reading the bell's
  permission state, the session's `sessionMachines` map, etc. Sparklines are
  re-attached every call (cache spares the HTTP, not the DOM rebuild).
- **Event listeners on agent-tree list items and session cards accumulate.**
  Both `buildAgentLi` and `buildSessionCard` call `addEventListener` on
  every rebuild. `updateOverviewGrid` rebuilds on every `updateSessions`;
  `renderAgentTree` replaces children but re-attaches. Net effect over a
  long-running dashboard is measurable DOM-listener leak.
- **The file has 94 `document.getElementById`/`addEventListener` call
  sites** spread across top-level code, IIFEs, and render paths. Initial
  DOM queries are repeated each render.

A clean reorganization would split into `router.js`, `ws-dispatcher.js`,
`state.js` (typed store with reactive update), `views/switchboard.js`,
`views/session-detail.js`, `views/history.js`, and `renderers/transcript.js`.
Details below.

## Correctness Bugs

### CRITICAL — Direct-URL load to session detail shows "not known" and never recovers

- **Location:** `dashboard.js:122-169` (`applyRoute`), `:482-489`
  (`updateSessions`), `:1928` (initial route apply), `:2216` (`connect()`).
- **Symptom:** Navigating directly to `https://claude.argo:3400/session/partyline-dev`
  paints "Session 'partyline-dev' is not currently known to the dashboard."
  even though that session is alive. Clicking the card from Switchboard to
  the exact same URL works.
- **Root cause:** Order of operations at load:
  1. Module evaluates, globals initialized to `lastSessions = []`.
  2. `applyRoute(parseUrl(), { skipPush: true })` runs at `:1928`.
  3. Since `state.view === 'session-detail'`, line `:131` computes
     `known = Array.isArray(lastSessions) && lastSessions.some(...)` — false,
     because the sessions list hasn't been populated.
  4. The `!known` branch at `:148` schedules a `setTimeout(..., 100)` that
     replaces the detail stream with the "not known" message.
  5. `connect()` runs at `:2216`, WS opens, eventually sends `sessions`.
  6. `updateSessions` → `updateOverviewGrid` paints the Switchboard cards.
     **But there is no re-check of the current route** — the stale "not
     known" DOM stays put forever.
     When clicking from Switchboard, `lastSessions` is already populated by
     the time `navigate()` → `applyRoute` runs, so `known` is true and the
     normal load path runs.
- **Fix sketch:**
  1. Track route state in a global (`currentRoute`) that `applyRoute`
     updates.
  2. In `updateSessions`, after the list is applied, check if
     `currentRoute.view === 'session-detail'` and the current
     `selectedSessionId` is now in `lastSessions` AND the stream is
     currently showing the "not known" placeholder — if so, re-run
     `applyRoute(currentRoute, { skipPush: true })` or call
     `loadSessionDetailView()`.
  3. Even simpler: in `applyRoute`'s session-detail branch, if `!known`,
     defer the "not known" message for _longer_ (say, 2-3s) AND install a
     one-shot listener on the next `sessions` payload that retries the
     known-check. Clear the timeout + listener when navigating away.
  4. Long-term: block route application behind a `sessionsReady` promise
     that resolves on the first `sessions` WS payload (see Architecture
     section).

### IMPORTANT — `applyRoute` has a TDZ workaround that silently swallows notifications

- **Location:** `dashboard.js:137-141`
  ```js
  try {
    notif.dispatchSessionViewed(state.sessionName)
  } catch {
    /* notif not yet initialized */
  }
  ```
- **Symptom:** On direct-URL load, the "mark session viewed" signal is
  silently dropped; notifications for that session won't be dismissed
  server-side on initial load.
- **Root cause:** `notif` is created at `:1963` (after `applyRoute` call at
  `:1928`), `const` declarations are in the TDZ during module init, so
  referencing `notif` throws `ReferenceError`. The `try/catch` hides the
  throw, but also hides a real bug.
- **Fix sketch:** Move the `const notif = createNotifications(...)` block
  _above_ `applyRoute(parseUrl(), ...)`. The factory doesn't depend on
  anything that runs later. Or, structurally: wire all initialization in
  an explicit `init()` function called once at the end.

### IMPORTANT — `selectedSessionId` vs `session.session_id` type confusion

- **Location:** `dashboard.js:876-898` (`handleSessionUpdate`), `:1133-1138`
  (`updateDetailHeader`).
- **Symptom:** `handleSessionUpdate` has two overlapping conditions:
  ```js
  if (currentView === 'session-detail' && selectedSessionId === session.session_id)
    updateDetailHeader(session) // uses DB UUID
  if (currentView === 'session-detail' && session.name === selectedSessionId)
    renderDetailHeader(session) // uses name
  ```
  `selectedSessionId` is **always a session name** (populated by the router
  from the URL). The first condition (comparing against `session.session_id`,
  a UUID) is dead code. `updateDetailHeader` will never fire from this path.
- **Root cause:** Historical drift — earlier versions keyed the session
  detail view by DB session ID, later moved to name. The old code wasn't
  removed.
- **Fix sketch:** Delete the `session.session_id` branch and document that
  `selectedSessionId` is a session name. Or, more correctly, introduce an
  explicit `selectedSessionName` variable and use it consistently.

### IMPORTANT — Stream-reset matches by UUID but `selectedSessionId` is a name

- **Location:** `dashboard.js:920-927` (`handleStreamReset`).
- **Symptom:** Compaction on remote host won't force a full transcript
  re-fetch when viewing the session.
- **Root cause:**
  ```js
  if (sessionName !== selectedSessionId && sessionId !== selectedSessionId) return
  ```
  `sessionId` is a UUID from the JSONL path; `selectedSessionId` is a
  session _name_. The comparison `sessionId !== selectedSessionId` is
  almost always true, so the only gate that works is `sessionName`
  (resolved via `resolveNameFromJsonlPath`, which requires the session to
  already exist in `lastSessions` with a matching `status.sessionId`). If
  the session hasn't announced itself yet, or the JSONL name-resolution
  fails, stream-reset is silently dropped.
- **Fix sketch:** Keep both a `selectedSessionName` and a derived
  `selectedSessionId` (UUID) in state, updated together whenever the
  `sessions` list changes or the session sends a session-update.

### IMPORTANT — `maybeHandleCompactForCurrentView` has the same name/UUID mismatch

- **Location:** `dashboard.js:937-952`.
- **Symptom:** SessionStart compaction events intended to force a re-fetch
  may match incorrectly or not at all.
- **Root cause:** Comments at `:945-946` say "Match by session UUID or
  session name (whichever the event carries)" — but both `session_id` and
  `session_name` are compared against `selectedSessionId` which is always
  a name. Same root cause as above.
- **Fix sketch:** Same as above; use an explicit UUID<->name map.

### NIT — `urlForView` emits `/history` vs `/history/events` inconsistency

- **Location:** `dashboard.js:100-105`.
- **Symptom:** Switching from Events subtab to Bus and back pushes
  `/history/bus` → `/history` rather than `/history/events`. Minor URL
  inconsistency; reload from `/history` lands on `events` (the default)
  so it works, but bookmarkability is uneven.
- **Fix sketch:** Always encode the subtab explicitly: either
  `/history/events` and `/history/bus`, or treat `/history` as
  "no subtab" and only go to it from the tab selector.

### NIT — `historySubtabs` click handler uses `replaceState`, not `pushState`

- **Location:** `dashboard.js:1853-1865`.
- **Symptom:** User navigates Events → Bus → Events; pressing Back goes to
  the previous view (not previous subtab). Inconsistent with other
  navigation which uses `pushState`.
- **Fix sketch:** Either normalize to `pushState`, or decide subtabs are
  intentionally non-navigable and document it.

### NIT — `addMessageToBus` is a no-op that adds WS dispatch noise

- **Location:** `dashboard.js:1849-1851`.
- **Symptom:** Dead function called once per party-line message
  (`:255`). Harmless but confusing.
- **Fix sketch:** Delete.

### NIT — `onsubmit="..."` inline handlers paper over module-scope isolation

- **Location:** `index.html:135-138, :172-175`, `dashboard.js:1959-1961`
  (`window.doBusSend = doBusSend`, `window.doDetailSend = doDetailSend`).
- **Symptom:** ES module globals leak onto `window` so inline `onsubmit`
  can find them. CSP-unfriendly; obscures wiring.
- **Fix sketch:** Convert forms to proper `.addEventListener('submit', ...)`
  handlers wired in JS; drop the window exports.

## Performance Issues

### IMPORTANT — `updateSessions(lastSessions)` is a full grid rebuild called from 6 sites

- **Location:** `dashboard.js:56, 214, 270, 403, 700, 2031`.
- **Symptom:** Every call visits every session, rebuilds every card's
  header+body+bell, fetches+renders every sparkline (cache-hit path still
  sets `innerHTML`), and re-runs bell permission logic. For N sessions,
  that's O(N) DOM work every time _any_ session-derived state changes —
  unread counts, context overrides, notification banner click, etc.
- **Root cause:** No reactive or selective update path. "Something
  changed, re-paint everything" is the default.
- **Fix sketch:**
  - Diff-mode: have `updateSessions` take a _delta_ (which session names
    changed, and in what dimension — state, unread, bell, ctx).
  - Event-based: subscribe individual card components to the specific
    pieces of state they render; re-render only affected cards.
  - Cheap interim: cache `buildCardContents`'s output and compare — if the
    serialized card equals the existing one, skip the replace.

### IMPORTANT — Every card rebuild calls `fetchAndRenderSparkline`, which always runs `innerHTML = ...`

- **Location:** `dashboard.js:623-641, 820-821, 846-847`.
- **Symptom:** On a cache hit (most common case), `innerHTML =
renderSparkline(buckets)` re-parses a 5-line SVG string, destroys any
  existing SVG, rebuilds. On N sessions \* M updates per minute, this is
  hundreds of needless SVG reparse cycles.
- **Root cause:** No "only re-render if changed" guard; cache keyed on
  freshness, not content identity.
- **Fix sketch:**
  - Skip if the current `sparklineSlot.firstChild` is already an SVG and
    the cached buckets hash hasn't changed.
  - Build the SVG as DOM nodes rather than innerHTML-ing a string.

### IMPORTANT — `renderAgentTree` wipes + rebuilds the whole subagent sidebar on every `session-update`

- **Location:** `dashboard.js:886-893` (fetch + `renderAgentTree()` on every
  session-update), `:1225-1282`.
- **Symptom:** `ul.replaceChildren()` then a loop appending LIs each with
  a fresh `addEventListener`. Happens on every session-update WS frame for
  the viewed session. If the user had a `<details>` group expanded, state
  may survive depending on exact DOM rebuild path — but it was still O(N)
  listener-add plus full repaint.
- **Root cause:** Coarse invalidation.
- **Fix sketch:** Compute the diff between `currentSessionSubagents` and
  the new list; only patch rows that changed. Use event delegation on the
  parent `ul` so the listener count is O(1) not O(N).

### IMPORTANT — `session-update` triggers both a `/api/session` refetch AND a `renderStream({incremental})` refetch

- **Location:** `dashboard.js:884-897`.
- **Symptom:** Every DB-aggregator tick (every time the server writes a
  session row — can be per-tool-call during bursty tool use) → two HTTP
  round-trips from every open dashboard tab. For a working session at
  "tool use every 2 seconds," this is ~30 req/min per tab.
- **Root cause:** Treating every session-update as "something changed,
  refetch everything."
- **Fix sketch:**
  - Debounce/coalesce session-update handling (e.g. 250ms trailing).
  - Diff incoming `session` payload against last known; refetch only the
    sub-resource that changed.
  - Many session-updates carry enough info to patch the detail header
    directly — no HTTP needed.

### IMPORTANT — `handleJsonlEvent` fires `renderStream({incremental:true})` for **every** JSONL line

- **Location:** `dashboard.js:900-912`.
- **Symptom:** Each JSONL line appended by the observer → WS frame →
  `renderStream` → `GET /api/transcript?after_uuid=...`. One HTTP per
  line. Observer polls every 1s so in practice this batches a bit, but
  under heavy tool-response output, it's request-per-tick.
- **Root cause:** No batching layer.
- **Fix sketch:** Debounce `renderStream({incremental})` with a 150-250ms
  trailing timer; after a burst, one fetch covers the whole delta.

### IMPORTANT — `seedUnreadCounts` fires N sequential awaited fetches on first session list

- **Location:** `dashboard.js:683-701`.
- **Symptom:** For N sessions, N serial `await fetch('/api/events?...')`
  calls. On a machine with 5 sessions that's 5\*RTT before the unread
  badges settle. Each returns 500 events (~50KB) which are then
  JSON.parsed and iterated client-side.
- **Root cause:** Sequential await in a for-loop; missing server-side
  aggregation endpoint.
- **Fix sketch:**
  - Add `GET /api/unread-counts?since_map=...` that returns
    `{session: count}` in one call.
  - At minimum, parallelize: `Promise.all(lastSessions.map(...))`.
  - Reduce the client-side filter (only the count is used) — ask the
    server for `count>since`, not the full event list.

### IMPORTANT — `.dataset.search = JSON.stringify(ev).toLowerCase()` is O(payload_size) per event

- **Location:** `dashboard.js:1724`.
- **Symptom:** For each history event, the entire payload is serialized
  into a DOM attribute. 500 events \* even 1KB avg payload = 500KB of data
  attributes. Filter keystroke → 500 `indexOf` scans of this full
  serialized corpus.
- **Root cause:** Simple-but-naive full-text filter.
- **Fix sketch:** Keep the serialized form in a JS Array indexed by DOM
  element, not on the DOM itself. Or compute filter match lazily and
  cache by event id. For real filter perf, build a proper searchable
  index.

### IMPORTANT — `list.insertBefore(li, list.firstChild)` on every hook-event WS frame reflows the whole history list

- **Location:** `dashboard.js:1788-1802`.
- **Symptom:** At 500+ history items, inserting at head + calling
  `applyHistoryFilters()` (which iterates all 500 items toggling
  classList) on every WS frame is measurable jank.
- **Root cause:** No buffer or virtualization.
- **Fix sketch:** Virtualize (render only visible + small buffer). At
  minimum: (a) skip `applyHistoryFilters` if no filter set; (b)
  requestAnimationFrame batch new inserts.

### NIT — `sparkline` uses polyline string-math; could use a pre-computed path

- **Location:** `dashboard.js:593-621`.
- **Symptom:** Minor — `toFixed(1)` + string concat. Low impact but O(N)
  allocation per sparkline render.
- **Fix sketch:** Build once, clone via `cloneNode` with attribute
  override.

### NIT — `addMessage` recomputes `new Date().toLocaleTimeString()` per frame

- **Location:** `dashboard.js:499`.
- **Symptom:** Minor. Under heavy heartbeat traffic, `Intl` formatting
  allocates.
- **Fix sketch:** Use a cached `Intl.DateTimeFormat` instance.

### NIT — `esc()` creates a throwaway `<span>` for each call but it's barely used

- **Location:** `dashboard.js:229-233`.
- **Symptom:** Dead-ish helper — no current call sites in the grep I did;
  if retained, it allocates a span per call rather than just using
  `.replace` on the 5 XML entities.
- **Fix sketch:** Verify no callers; delete. Or replace with the standard
  5-char replace.

## State Management Hazards

### CRITICAL — Twelve module-level mutable globals, no single source of truth

- **Location:** `dashboard.js:9-16, 50, 76-79, 1308-1314, 1715-1717`.
- **List:** `totalMessages`, `ws`, `contextOverrides`, `lastSessions`,
  `sessionSources`, `currentView`, `localMachineId`, `sessionMachines`,
  `selectedSessionId`, `selectedAgentId`, `currentSessionSubagents`,
  `historyBuffer`, `unreadCounts`, `seededOnce`, `renderedEntryKeys`,
  `renderedStreamKey`, `missedWhileScrolledUp`, `lastRenderedUuid`,
  `historyEvents`, `historyHooksSeen`, `historyLoaded`, `sparklineCache`.
- **Symptom:** Any code path can mutate any of these. `updateSessions`
  reads `unreadCounts`; `handleSessionUpdate` writes `sessionSources`,
  `sessionMachines`, `currentSessionSubagents`, and fires further render
  effects; `renderStream` mutates `renderedEntryKeys`, `renderedStreamKey`,
  `lastRenderedUuid`, `missedWhileScrolledUp`. No invariants enforced.
- **Risk:** Any future bug that leaves e.g. `selectedSessionId` set after
  switching to Switchboard means subsequent `session-update` frames try
  to patch a view that isn't visible, which isn't a crash but paints
  invisible DOM.
- **Fix sketch:** Consolidate into a `state` object with typed accessors,
  or move to a tiny observable store (Proxy-based or a 20-line
  publish/subscribe). See Architecture.

### IMPORTANT — `selectedSessionId` is a _name_, while some WS payloads carry _UUID_ — no single source of truth

- **Location:** `dashboard.js:876, 920, 937`.
- **Symptom:** See Correctness Bugs above. The `session_id` vs
  `session_name` vs `name` distinctions are mixed throughout the file and
  the comments in the code acknowledge the confusion (`// session here
is an aggregated DB session with session_id, not the heartbeat
session`, `:868-869`).
- **Root cause:** Organic growth from a name-based world to a hybrid
  name+UUID world without a refactor.
- **Fix sketch:** Introduce a typed `SessionRef = { name: string;
sessionId?: string; machineId?: string; source?: string }` and use it
  everywhere. A `resolveRef(refOrName)` helper looks up missing fields
  against the global sessions map.

### IMPORTANT — `sessionSources` / `sessionMachines` only populated on `session-update`

- **Location:** `dashboard.js:871-877`.
- **Symptom:** A session that shows up via multicast but hasn't yet
  produced a DB session-update row (no hook events) renders without the
  Gemini/host badge. The badges appear when the first session-update
  arrives — sometimes minutes later.
- **Root cause:** Two different event streams populate overlapping state.
  Multicast presence knows name+host; DB aggregator knows name+source.
  The client merges them by ad-hoc globals.
- **Fix sketch:** Have the server include `source` + `machine_id` on the
  multicast-derived `sessions` payload too.

### IMPORTANT — `unreadCounts[s.name] = 0` on view but not on 'session-viewed' WS dispatch path

- **Location:** `dashboard.js:212-214, 44-46`.
- **Symptom:** Opening the detail view of session `foo` zeros
  `unreadCounts['foo']` locally — but if the same user has the dashboard
  open in two tabs, the other tab won't know to zero its unread badge
  (localStorage is per-origin but `updateSessions` only runs on the
  writing tab via its own code path).
- **Root cause:** localStorage-based state + no `storage` event handler.
- **Fix sketch:** Listen for `storage` events and sync across tabs.
  Alternatively, have the server broadcast a `session-viewed` message to
  all WS clients (ties to the one-way `dispatchSessionViewed` already
  present, `notifications.js:179-182`).

### IMPORTANT — `renderedEntryKeys` and `lastRenderedUuid` can desync on concurrent `renderStream` calls

- **Location:** `dashboard.js:1308-1314, 1347-1451`.
- **Symptom:** `renderStream` is async (`await fetch(...)`). If
  session-update and JSONL-event arrive within the same tick, both can
  start overlapping `renderStream` calls. The second call sees the first
  call's mid-flight state.
- **Root cause:** Race — no lock / no AbortController / no monotonic
  request ID.
- **Fix sketch:** Single in-flight `renderStream` promise; if called
  while one is pending, set a "redo" flag and chain it. Or use
  `AbortController` and explicit cancellation.

### NIT — `historyLoaded` is a one-way latch, never reset on disconnect/reconnect

- **Location:** `dashboard.js:1717, 1788-1802`.
- **Symptom:** After WS reconnect, history list is stale (shows what was
  cached from the first load) until user navigates away and back.
- **Fix sketch:** Reset `historyLoaded = false` on `ws.onclose`.

### NIT — `sparklineCache` uses `sessionId` which is sometimes `session.name`, sometimes `sessionId`

- **Location:** `dashboard.js:816-821, 591-641`.
- **Symptom:** Comment at `:816-817` admits "Use session name as
  session_id proxy for party-line sessions." Cache entries can collide
  or miss across a session's DB-row transition.
- **Fix sketch:** Namespace cache keys: `sparklineCache[`name:${name}`]`
  vs `sparklineCache[`uuid:${id}`]`.

## Realtime Inconsistency

### IMPORTANT — Incoming party-line message takes a fundamentally different path than a session-update

- **Location:** `dashboard.js:253-266` (message handler), `:271-274`
  (session-update handler).
- **Symptom:** When a message arrives:
  - Switchboard: `bumpUnread` → `updateSessions(lastSessions)` → full
    grid rebuild. But the card's `last-text` didn't change (multicast
    messages don't patch the session status).
  - Session detail: `appendEnvelopeToStream(data.data)` appends one DOM
    node.
    When a session-update arrives:
  - Switchboard: `updateSessions(lastSessions)` is NOT called → the card
    doesn't update. `bumpUnread(data.data.name)` DOES call it, but the
    card pulls state from `lastSessions` which didn't change.
  - Session detail: `renderDetailHeader` + `/api/session` refetch +
    `renderStream({incremental})`.
    Net: the Switchboard card's `state` pill (`working`/`idle`) updates
    from multicast heartbeats (every N seconds) but NOT from the much-more-
    frequent DB session-update events. User sees the card as "working" for
    tens of seconds after Claude has gone idle.
- **Root cause:** `handleSessionUpdate` only touches the detail view.
  The card update is left to the next multicast tick.
- **Fix sketch:** `handleSessionUpdate` should also patch
  `lastSessions[i].metadata.status` for the matching session and call
  `updateOverviewGrid` (or, better, a card-only targeted update).

### IMPORTANT — `appendEnvelopeToStream` assumes dedup by `envelope.id`, but incremental `renderStream` dedups by `e.uuid`

- **Location:** `dashboard.js:956-999`, `:1428-1436`.
- **Symptom:** A message envelope is appended to the stream with
  `renderedEntryKeys.add(envelope.id)`. Later when the JSONL observer
  sees the recipient's user-turn (the incoming message recorded in the
  JSONL), `renderStream` fetches it; its uuid is the JSONL UUID, not the
  envelope UUID. Dedup misses → the same message renders twice.
- **Root cause:** Two different keys for the "same" logical entry.
- **Fix sketch:** Include envelope_id in the transcript entry so the
  server returns it; dedup on either.

### IMPORTANT — `cross-call` overlay only works when `currentView === 'switchboard'`

- **Location:** `dashboard.js:297-331`.
- **Symptom:** Switching to Session Detail silences the cross-call
  animation layer. When user returns to Switchboard, in-flight arrows
  don't catch up; it feels like arrows are "missing."
- **Root cause:** Animation tied to currently-visible DOM only.
- **Fix sketch:** Either defer-and-replay a ring buffer of recent calls
  on view change, or accept this is intentional and make clear (no
  cross-calls while in detail view). Document it.

### IMPORTANT — `showHeartbeats` / `showAnnounce` checkboxes are read but never defined in `index.html`

- **Location:** `dashboard.js:5-7, 496-497`, `index.html` (grep confirms
  no `id="showHeartbeats"` element).
- **Symptom:** Both `const` handles are `null`. The `msg.type ===
'heartbeat' && showHeartbeats && !showHeartbeats.checked` check
  short-circuits to false → all heartbeats end up in the Bus feed.
  Totals balloon, scroll-to-bottom thrashes, Bus feed is unusable during
  normal operation.
- **Root cause:** Orphaned controls from an earlier index.html.
- **Fix sketch:** Either add the checkboxes to the Bus subtab's feed
  controls, or hardcode the filters (exclude heartbeat/announce from Bus
  by default and surface a toggle).

### NIT — Detail view bell permissions read-through is stale

- **Location:** `dashboard.js:790-801`, `:1210-1223`.
- **Symptom:** `buildBellButton` caches `notif.getPermissionState()` at
  card-build time. If user grants permission while the card is
  mounted, the bell stays disabled until next full repaint.
- **Fix sketch:** When permission state flips, force a single
  `updateSessions(lastSessions)` refresh. Already done in one spot
  (`:2029-2031`) — apply consistently.

## Architectural Recommendations

Current file: 2,216 lines, single module, 22 module-scope globals, 6
distinct concerns (router, WS dispatcher, renderer, state, notifications,
history), all entangled. Proposed layout:

```
dashboard/
  client/                         # new — all browser code
    index.js                      # bootstrap: init state, connect WS, apply route
    router.js                     # parseUrl, urlForView, pushRoute, applyRoute
    store.js                      # typed state store + subscribe()
    ws.js                         # connect(), reconnect, frame dispatcher
    api.js                        # typed fetch wrappers (getSession, getTranscript, …)
    views/
      switchboard.js              # grid + card rendering
      session-detail.js           # header, agent tree, stream
      history.js                  # events subtab, bus subtab
    renderers/
      transcript.js               # renderEntry + renderMarkdownInto
      sparkline.js                # renderSparkline + caching
      permission-card.js          # renderPermissionCard + respondToPermission
```

### 1. State store with observers

Single source of truth — a `store.js` exposing:

```js
// pseudo-TypeScript
type SessionRef = { name: string; sessionId?: string; machineId?: string; source?: string }
type AppState = {
  route: { view: 'switchboard'|'session-detail'|'history'; sessionName?: string; agentId?: string; subtab?: string };
  sessions: Map<string, SessionRef & { metadata: {...} }>;
  sessionsReady: boolean;            // true after first 'sessions' WS frame
  unreadCounts: Map<string, number>;
  contextOverrides: Map<string, { contextLimit: number }>;
  selected: { sessionName?: string; agentId?: string; subagents: Subagent[] };
  stream: { key: string; lastUuid: string | null; renderedKeys: Set<string> };
  history: { loaded: boolean; events: HookEvent[]; hooksSeen: Set<string> };
  quota?: QuotaStatus;
  localMachineId?: string;
}

export function subscribe(selector, handler) // re-invoke handler when selector output changes
export function dispatch(action)              // reducer-style
export function getState()
```

Views subscribe to the slice they need. Card components subscribe to
`sessions.get(name)` + `unreadCounts.get(name)`. The Switchboard no
longer rebuilds every card on every tick.

### 2. Explicit lifecycle & `sessionsReady` gate

`applyRoute` should **not** render "not known" synchronously. If the
router targets a session and `store.sessionsReady === false`, queue the
"unknown" fallback for _after_ `sessionsReady` becomes true. The
critical bug above disappears.

```js
function applyRoute(state) {
  if (state.view === 'session-detail') {
    if (!store.get().sessionsReady) {
      // Render loading state; retry once sessionsReady flips.
      renderSessionDetailLoading(state.sessionName)
      store.once('sessionsReady', () => applyRoute(state))
      return
    }
    // ...existing logic
  }
}
```

### 3. WebSocket dispatcher as a dumb pipe

`ws.js` should ONLY:

- Open/reconnect the socket.
- JSON.parse frames.
- `dispatch({ type: frame.type, payload: frame.data })`.

All business logic moves into reducers / subscribers. This makes
behaviour testable (feed fake frames, assert state transitions).

### 4. Debounced / batched update boundary

Add a `requestIdleCallback`-style scheduler between `dispatch` and view
re-render:

```js
let pending = false
function scheduleRender() {
  if (pending) return
  pending = true
  requestAnimationFrame(() => {
    pending = false
    emitter.emit('render')
  })
}
```

All WS frames → dispatch → schedule. At most one render per frame
regardless of WS frame rate. Kills the N-fetch-per-session-update
storm.

### 5. Typed transcript + name/UUID disambiguation

Introduce `SessionRef` type and a client-side map. Never let a bare
string cross a function boundary where it could be either a name or a
UUID.

### 6. Targeted rendering

`updateOverviewGrid` should compute which cards changed (which session
ids were added / removed / updated) and only touch those cards. For
updated cards, compute the specific fields that changed (state, ctx,
last-text, unread, bell) and update only those DOM nodes.

Simplest implementation: key-and-data diff. Each card stores the last
rendered data snapshot as a `WeakMap<Element, Snapshot>`; on update,
compare new snapshot to old and patch only the diff.

### 7. Delete-before-add: quick wins with minimal refactor

Even short of the full reorg above, the following are self-contained
cleanups that would noticeably improve reliability:

1. Move `const notif = createNotifications(...)` above
   `applyRoute(parseUrl())` — eliminates the TDZ try/catch.
2. Add `sessionsReady` gate and a `once-sessions-arrive` retry in
   `applyRoute` — fixes the CRITICAL direct-URL bug.
3. In `handleSessionUpdate`, also patch `lastSessions` and trigger a
   card update — fixes the "card pill stuck at working" issue.
4. Coalesce rapid-fire `session-update` and `jsonl` events through a
   250ms trailing debounce before firing `renderStream` — eliminates
   fetch storms.
5. Add a `GET /api/unread-counts` endpoint; replace the sequential
   per-session fetch in `seedUnreadCounts`.
6. Remove `showHeartbeats`/`showAnnounce` dead handles OR add the
   matching `<input>` elements to `index.html`.
7. Delete the dead `selectedSessionId === session.session_id` branch
   in `handleSessionUpdate`.
8. Use event delegation on `#detail-tree` and `#overview-grid` so
   listeners don't accumulate on rebuild.
