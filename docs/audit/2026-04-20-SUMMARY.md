# Audit Master Summary — 2026-04-20

Four read-only audits were run in parallel against the Claude Party Line codebase.
This doc consolidates their findings into a single prioritized remediation plan.

## Headline

The project is functionally coherent — the protocol round-trips, the observers
don't crash, the dashboard renders. Unit tests pass. But the foundation is
fragile in predictable places:

1. **Browser notifications don't work on mobile at all** and on desktop fall into
   the Chromium abuse-heuristic trap. Phase 1's "foreground-only, no Service
   Worker" design was wrong — `new Notification()` throws `TypeError: Illegal
constructor` on Chrome Android. The whole feature needs to rebuild on a
   Service Worker.
2. **Direct-URL loads to session detail are permanently broken** because
   `applyRoute` runs at module eval before the WS delivers `sessions`, and
   `updateSessions` never re-evaluates the route when the list later arrives.
3. **DB migrations never actually run on upgrade** — `schema.sql` contains
   `PRAGMA user_version = 3` which is set before the migration runner checks,
   so v1→v2 and v2→v3 never execute.
4. **Four unsynchronized update channels** (`hook-event`, aggregator
   `session-update`, `jsonl`, periodic `sessions`) race to refresh the same UI
   with no version/revision tag. This is the "realtime feels inconsistent"
   symptom.
5. **Zero end-to-end tests.** Every real bug the user hit maps directly to a
   coverage gap. The client's 2,200 LOC has no test harness at all.

## IMPORTANT findings (condensed)

- **N4** — `notif.onPartyLineMessage`/`onSessionUpdate` called without try/catch
  in the WS handler; one throw kills future frames' notifications.
  (`dashboard.js:257, 274`)
- **N5** — `activeNotifications.get(tag).close()` on cached instance won't work
  in SW model (needs `registration.getNotifications({ tag })`).
- **C3** — `selectedSessionId === session.session_id` is dead code;
  `selectedSessionId` is always a name. Also corrupts `handleStreamReset`.
- **C4** — `handleSessionUpdate` doesn't patch the Switchboard card; pill
  stuck at "working" until next heartbeat.
- **C5** — `appendEnvelopeToStream` dedups by `envelope.id` but `renderStream`
  dedups by JSONL `uuid` — same message can render twice.
- **C6** — `showHeartbeats`/`showAnnounce` handles referenced but no
  `<input>` in `index.html` → heartbeats spam the Bus feed.
- **C7** — Every `session-update` triggers `GET /api/session` +
  `/api/transcript` refetch. Bursty tool use → fetch storm.
- **C8** — Event listeners accumulate on every card/agent rebuild.
- **S6** — `insertEvent + aggregator.ingest` are three separate DB commits per
  PostToolUse; should be one transaction.
- **S7** — `OR session_name` in `events`/`tool_calls` queries disables
  indexes.
- **S8** — Permission-bridge pending map is unbounded + no TTL/timeout.
- **S9** — JSONL observer drops lines that straddle poll boundaries.
- **S10** — `hourlyToolCalls` scans events table every sparkline render.
- **S11** — Four update channels race without a revision tag. This is the
  "flicker" root cause.
- **T3** — `tests/server-permission.test.ts:9-12` is a text-grep, not a
  runtime assertion.
- **T4** — `src/introspect.ts` (333 LOC that drives every heartbeat) is
  entirely untested.
- **T5** — `src/transport/udp-multicast.ts` — the core product claim — has
  no tests.

## Proposed remediation plan (for your approval)

Four phases. Each phase ships before the next starts. Each produces working,
dogfood-able software.

### Phase A — Foundation fixes (estimated 1 day)

Smallest deltas that unblock everything else.

1. **Add `test` script to `package.json` + pre-commit + basic CI baseline**
   so nothing else regresses silently.
2. **Fix the migration runner** (S1) — remove `PRAGMA user_version = 3` from
   `schema.sql`, wrap each migration in a transaction, add an upgrade test.
3. **Fix the direct-URL bug** (C1) — introduce a `sessionsReady` gate; defer
   the "unknown" fallback until after the first `sessions` WS frame arrives.
4. **Move `const notif = createNotifications(...)` above `applyRoute`** — kills
   the TDZ try/catch nonsense.
5. **Delete or fix the dead/stale references** (C3, C6): remove the
   `session_id` comparison in `handleSessionUpdate`, either add the
   `<input>` elements for the heartbeat filters or delete the handles.
6. **Remove `PRAGMA user_version` from schema.sql.** (covered by #2 above but
   worth naming separately.)

### Phase B — Notifications rebuild (estimated 0.5 day)

Single self-contained area, addresses N1–N5 all at once.

1. Add `dashboard/sw.js` (~30 lines: install/activate + notificationclick
   with `clients.matchAll → focus/openWindow`).
2. Register in `dashboard/dashboard.js` at bootstrap
   (`navigator.serviceWorker.register('/sw.js')`).
3. Replace `new Notification(...)` in `notifications.js` with
   `(await navigator.serviceWorker.ready).showNotification(...)`.
4. Replace `activeNotifications.close()` with
   `registration.getNotifications({ tag })` lookups.
5. Remove the auto-showing banner. Trigger permission request only from an
   explicit click on the bell — and request permission synchronously inside
   the click handler (no `await` before the call).
6. Add `navigator.permissions.query({name:'notifications'}).onchange` listener
   - `visibilitychange`/`focus` re-read so banner state reflects reality.
7. Wrap `notif.onPartyLineMessage`/`onSessionUpdate`/etc. callers in
   `try/catch` in `dashboard.js` so one throw can't kill the rest of the WS
   handler.
8. Update tests to run against the SW-based API (can still mock
   `navigator.serviceWorker`).

### Phase C — Server foundation (estimated 1.5 days)

Fixes the "flickery realtime" root cause and the quieter correctness bugs.

1. **Unify update channels** (S11) — introduce `session-delta` WS frame with
   a monotonic `revision` per session. Client keeps a revision map and
   ignores stale frames. `sessions` becomes handshake-only. Removes the
   race between `hook-event`, `session-update`, `jsonl`, and periodic
   snapshots.
2. **Wrap `insertEvent + aggregator.ingest` in one transaction** (S6).
   Prepare statements at server startup.
3. **Remove `OR session_name` from hot query paths** (S7); resolve
   `session_key` → UUID once at ingest and index on UUID only.
4. **Fix dedup** (S2): either synchronous prune on `isDuplicate()` or LRU
   cap; align the prune interval and the TTL.
5. **Fix send-twice** (S3): await both sends; if the second fails log + emit
   a health warning.
6. **Remove `seq`** (S4) — dead wire weight. Simpler than retrofitting real
   per-peer sequencing, which nothing needs yet.
7. **Permission-bridge TTL** (S8) — bounded pending map with auto-deny after
   30s, logged.
8. **Machines table** (S5) — either write it correctly on heartbeat + expose
   it via `/api/machines`, or delete the table and the tab. Recommend write
   it; low effort, visible value.
9. **JSONL observer partial-line handling** (S9) — buffer unterminated
   bytes, don't advance offset until a `\n` is seen.

### Phase D — Client architecture (estimated 2 days)

The 2,216-line `dashboard.js` is the biggest single liability. This phase
is a careful split, not a rewrite.

1. Extract `dashboard/client/` directory with split modules
   (`router.js`, `store.js`, `ws.js`, `api.js`, `views/{switchboard,session-detail,history}.js`,
   `renderers/`, `services/`, `utils/`).
2. Introduce a typed state store with `subscribe(selector, handler)` so views
   re-render only on relevant changes.
3. Add a `requestAnimationFrame`-coalesced render scheduler between
   `dispatch` and render — kills fetch storms and DOM rebuild storms.
4. Event delegation for Switchboard grid + agent tree → fixes listener leak.
5. Replace `seedUnreadCounts` N-serial-fetch with `GET /api/unread-counts`
   (new endpoint) returning the map in one call.

### Phase E — Tests (estimated 2 days, runs partly in parallel with C/D)

1. MCP two-process conversation test (boots the real stdio server, sends
   tools/notifications, validates round-trip).
2. `serve.ts` in-process integration test (boot Bun.serve, connect a real
   WS client, walk the ingest → aggregator → broadcast path).
3. UDP multicast loop test (two `UdpMulticastTransport` instances, send
   both with and without `includeSelf`, assert dedup).
4. Client router state machine — extracted from `dashboard.js` Phase D and
   tested in isolation.
5. Playwright smoke spec with `context.grantPermissions(['notifications'])`
   exercising the SW path. One spec to start.

## Total estimated effort

**~7 days of focused work**, ordered A → B → C → D (with E running across
C/D). Parallelizable into two streams (client/notifications + server/tests)
if you want to split across sessions.

## Open decisions for the user

Before starting any work, decide:

1. **Are all five phases in scope?** Or do you want to ship A + B first,
   review state, then decide on C/D/E?
2. **Branching strategy?** One feature branch per phase with PRs to main?
   Or one `audit-remediation` long-lived branch?
3. **Subagent-driven execution?** The superpowers workflow (fresh
   implementer per task, two-stage review per task) fits well here because
   the phases are largely independent. Or straight inline with a single
   implementer.
4. **`seq` removal** — strictly a breaking wire change even though nothing
   uses it. Confirm before doing it.
5. **PWA / manifest / Push + VAPID** — the notifications rebuild (Phase B)
   deliberately stops short of these. Confirm that's still the right line.
