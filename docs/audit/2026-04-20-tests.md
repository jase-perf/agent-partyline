# Test Coverage + Dogfood Gaps Audit — 2026-04-20

## Summary

The project has ~850 lines of unit tests across 14 files, running on `bun:test`, covering
pure-module surface area reasonably well: protocol round-trips, envelope validation, SQLite
schema + queries, daily metrics rollup, aggregator state transitions, JSONL observer tailing
and truncation, ingest HTTP + auth, Gemini transcript observer, transcript builder (with
isMeta/compaction scenarios), permission-bridge logic, serve-helpers wire-framing, and the
notifications factory module with injected deps.

**But every test either (a) exercises a pure function with no I/O, or (b) exercises a module
in isolation with mocks.** There are **zero** tests that:

- Start the real MCP server via stdio and exchange tools/notifications with a second MCP
  session, or any end-to-end MCP handshake.
- Boot `dashboard/serve.ts`, connect a WebSocket client, and walk a real event path
  (ingest → aggregator → broadcast, or multicast → serve → WS frame → browser).
- Exercise `dashboard/dashboard.js` (~2,200 LOC of client JS, including the router,
  WebSocket message dispatch, view rendering, notification wiring) under jsdom, Playwright,
  or any DOM host.
- Send real UDP multicast datagrams between two `UdpMulticastTransport` instances.

There is also **no `test` script in `package.json`** — tests are discovered only if a
contributor knows to run `bun test`. CI-style execution isn't wired in.

The four real bugs the user hit map cleanly onto these gaps:

| Bug                                              | Why tests missed it                                                                                                                                                                                                                                                                                                                    |
| ------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Direct-URL `/session/:name` stuck on placeholder | No client-router test; no jsdom/Playwright harness for `dashboard.js`. The bug is a WS→router ordering race; unit tests of `createNotifications` can't see it.                                                                                                                                                                         |
| Foreground-only notifications on mobile          | `notifications.test.ts` runs against an in-process `FakeNotification` with `permission = 'granted'` by default. It never tests actual browser behavior on a tab that is backgrounded but still "visible" to the OS, nor mobile Chrome's requirement that background notifications come through a Service Worker.                       |
| Banner Enable button no-op / auto-deny           | No test for `updateBanner()` or the click handler in `dashboard.js`. No test that verifies "clicks on the banner button happen inside a real user gesture" or that repeated dismissed prompts trigger the abuse heuristic.                                                                                                             |
| Dashboard-as-recipient missed in dogfood         | Was added reactively in `tests/notifications.test.ts` lines 351–395 after the bug was seen. Pre-fix, no test exercised the "agent sends to=dashboard and the dashboard is the listening endpoint" path. The `PartyLineMonitor` never surfaced `to === 'dashboard'` frames as anything special because no test ever posed the question. |

## Tests That Don't Assert Meaningfully

### IMPORTANT — server capability test is a text-file grep

- File: `tests/server-permission.test.ts:9-12`
- What it claims: "server.ts declares claude/channel/permission capability".
- What it does: `readFileSync(server.ts)` then `expect(source).toContain("'claude/channel/permission': {}")`.
- Why it's weak: This is a text-file assertion. It tells you the literal string is present;
  it does not tell you the MCP server actually advertises the capability over the wire. Any
  refactor that builds the capabilities object via a spread or a conditional will pass the
  grep but could regress the runtime behavior. Vice versa, a typo that breaks the object
  literal (`'claude/channel/permission':{}` with no space, or a missing quote) would flip this
  test red without any real regression.
- Suggested change: Boot the MCP server via stdio transport, send a `initialize` request,
  and assert the returned `capabilities.experimental['claude/channel/permission']` exists.

### IMPORTANT — dedup "first-ever update records state but does not fire"

- File: `tests/notifications.test.ts:91-98`
- Assertion: first update doesn't fire a notification.
- Why it's weak: the test leaves `ctx.doc.hidden = true` and `setEnabled('research', true)`,
  then calls `onSessionUpdate({ state: 'idle' })` and expects 0 fires. This is trivially
  true — nothing transitioned from working→idle, so the fire path is never reachable. The
  test doesn't actually prove that "first-ever" is handled specially; it just proves that
  `idle` (not preceded by `working`) doesn't fire.
- Suggested change: Either delete it or make it stronger by sending `idle` first, then a
  `working` update, and asserting still 0 fires (because working→idle hasn't happened yet).

### IMPORTANT — mocks that never fail the promise paths

- File: `tests/notifications.test.ts:319-332` ("Trigger A fetches transcript and uses last
  assistant text as body")
- The fetch mock always returns `ok: true` and the injected `setFetchResponse`. There's no
  test for `ok: false` (server 500 on `/api/transcript`), no test for non-array JSON (e.g.
  `{error: "..."}`), no test for empty transcript (no assistant-text entries). The code path
  `if (Array.isArray(entries))` at notifications.js:112 is one branch; `if (res.ok)` at :110
  is another; neither is negatively tested, so regressions to "falls through to generic body
  even on a success response with no assistant text" wouldn't be caught.
- Suggested change: Add tests for `ok: false`, `json returns {error}`, transcript contains
  only `user` and `tool-use` entries (no `assistant-text`).

### IMPORTANT — ingest-http 500 test relies on closed-db as stand-in

- File: `tests/ingest-http.test.ts:68-82`
- Asserts status 500 when storage fails by closing the DB before dispatching the request.
  The real failure mode in production is more likely "db is full / disk I/O error / FK
  constraint" not "db has been explicitly closed." The test proves the error branch returns
  500 but doesn't exercise the typical failure shape.
- NIT — fine for a smoke test but worth noting: errors are swallowed silently in production
  (no logging path is tested), and the test doesn't assert the response body.

### NIT — storage "idempotent" test asserts nothing

- File: `tests/storage.test.ts:28-32`
- The test opens+closes a DB, then opens it again, closes it. It has zero `expect()` calls.
  It passes if openDb doesn't throw. That's a valid smoke-test intent, but it's invisible as
  a meaningful assertion to a reader and doesn't defend against a future `openDb` that leaks
  state across opens.
- Suggested change: Add `expect(tables).toContain('machines')` after the second open, or
  open a third time and assert schema version is still 3.

### NIT — jsonl-observer "handles large appended chunks"

- File: `tests/jsonl-observer.test.ts:102-122`
- Writes 10 \* 5000-byte lines (50kB) and expects 10 events. Good test in principle.
  But "large" here is tiny. A real compaction can append megabytes. At minimum the test
  should verify there's no re-entrancy bug if the file grows again mid-poll. Consider
  adding a test that appends another chunk before the first poll completes.

### NIT — aggregator listener fires are not scoped

- File: `tests/aggregator.test.ts:118-130`
- The listener test asserts the listener fired once with the expected session_id. It doesn't
  assert that unrelated session ingests don't fire the listener, nor that multiple listeners
  can co-exist. The production `onUpdate` fires for every ingest — that's an N^2 broadcast
  risk that isn't tested against.

### NIT — event validation covers positive + simple negatives but not envelopes

- File: `tests/events.test.ts`
- Good coverage of scalar field validation. Missing: very large payloads, deeply nested
  payloads, event with `null` fields where strings expected, Unicode / emoji in
  session_name, extremely long session_name (>256 chars).

### NIT — transcript test uses real-wall-clock timestamps

- File: `tests/transcript.test.ts:217-220` (after_uuid envelope cutoff test)
- Uses `new Date(Date.now() + 60_000).toISOString()` for future envelope timestamps to make
  them sort after JSONL entries. This works but couples the test to the fact that
  `buildTranscript` assigns `Date.now()` when a JSONL entry has no `timestamp` field. A
  refactor that uses a fixed epoch for missing timestamps would break this test for the
  wrong reason.

## Integration / Dogfood Gaps

These scenarios have **no automated coverage** and have all produced regressions in dogfood:

1. **Two MCP sessions actually messaging each other.** There is no test that spawns two
   `bun src/server.ts` processes, has one send a `party_line_send` tool call, and asserts
   the other receives a `notifications/claude/channel` MCP notification. This is the
   entire point of the project. The test suite passes even if `server.ts` has never been
   successfully run.

2. **Full multicast loop with dedup.** No test creates two `UdpMulticastTransport`
   instances, sends an envelope from A, asserts B receives it exactly once (because of
   send-twice + dedup), and B doesn't receive its own sends (unless `includeSelf=true`).

3. **Dashboard boot → WS connect → load `/session/:name` directly.** No test starts
   `dashboard/serve.ts`, drives a headless browser to `http://localhost:PORT/session/foo`,
   waits for sessions WS frame, and asserts the session detail pane renders (this is the
   exact failure the user reported — the client router resolves the URL before WS delivers
   `lastSessions`, so the "not currently known" placeholder sticks and never recovers when
   `lastSessions` arrives because `updateSessions()` doesn't re-apply the route).

4. **Browser-context notification fire.** Every notifications test uses `FakeNotification`,
   which means the test asserts the module's _logic_ for when to construct a Notification
   instance. It does **not** assert that a real browser can actually display the
   notification on a backgrounded tab, on mobile, under an insecure context (HTTP over LAN
   IP), in a Service-Worker-required mobile scenario. The production bug on mobile Chrome
   almost certainly requires a Service Worker + `registration.showNotification()` path —
   which doesn't exist in code and isn't flagged by any test.

5. **Permission-request round-trip through the real UDP bus.** The end-to-end test at
   `tests/notifications-integration.test.ts` simulates every hop with inline function calls.
   It doesn't send a real multicast datagram. The day the transport sends the permission
   envelope differently (e.g., chunked, multi-destination, wrong group) the notifications
   integration test still passes.

6. **Hook emitter → /ingest → aggregator → WS broadcast.** There's a test for `/ingest`
   (good) and a test for aggregator (good), but nothing plumbs the full path including the
   `onEvent` callback hooked into `serve.ts`'s WebSocket broadcast. A regression in how
   `serve.ts` wires `onEvent` would pass both sub-tests and still break the dashboard.

7. **Stream-reset → forced client re-fetch.** Observer emits reset, tested. Serve.ts
   turning the reset into a `stream-reset` WS frame, tested (implicit). The client-side
   `handleStreamReset` in `dashboard.js`, untested.

8. **Direct-URL deep links with subagent ID (`/session/foo/agent/abc`).** Router code at
   `dashboard.js:86-97` parses this shape. No test.

9. **Banner button click behavior including repeated clicks.** No test — and this is the
   exact vector for Chrome's abuse heuristic.

10. **WebSocket reconnect behavior.** `dashboard.js:244-248` retries on close. No test of
    the reconnect flow, state restoration (do unread counts survive? does selectedSessionId
    survive? do we re-fetch history?).

11. **"dashboard is the recipient" on the live bus.** `notifications.test.ts:351-395` was
    added to test the notification surface, but there is no end-to-end path test from
    `PartyLineMonitor` receiving the `to='dashboard'` envelope to the WS frame the client
    sees. If a future refactor in `dashboard/serve.ts:136-152` (cross-call emission)
    silently filters `to='dashboard'`, nothing will catch it.

## Recommended Tests to Add

For each of the four real bugs in the audit context, here is the concrete test to add,
where it should live, and how it would have failed before the fix.

### Bug 1: Direct URL `/session/<name>` stuck on placeholder

- File: `tests/client-router.test.ts` (new — requires jsdom or Playwright)
- Depends on: adopting either `happy-dom`/`jsdom` (cheap) or Playwright (more realistic).
- What it exercises: load `dashboard.js` into a DOM, set `window.location.pathname =
'/session/research'`, trigger `applyRoute(parseUrl(), { skipPush: true })` with
  `lastSessions = []`, assert placeholder is shown. Then dispatch a
  `{type:'sessions', data:[{name:'research',...}]}` WS message and assert the placeholder
  is replaced by real content.
- Pre-fix failure mode: After the sessions frame arrives, the DOM still shows the
  "Session 'research' is not currently known" text because `updateSessions()` at
  `dashboard.js:482-489` does not re-apply the current route. The test would fail at the
  second assertion.

Alternative unit-level test if DOM host is too heavy:

- File: `tests/client-router-logic.test.ts` — extract `applyRoute` / `updateSessions` into a
  pure function or expose a "re-resolve current route" hook. Assert calling
  `updateSessions(sessionsList)` while `currentRoute.view === 'session-detail' &&
sessionName not in previous sessions` causes the route to re-resolve.
- Pre-fix: the extracted unit has no code path to re-apply → test fails immediately.

### Bug 2: Foreground-only notifications on mobile Chrome

- File: `tests/notifications-service-worker.test.ts` (new) + `tests/mobile-e2e.spec.ts`
  (new, Playwright).
- Unit test: once a Service-Worker-based path is introduced, test that
  `createNotifications` calls `registration.showNotification(title, options)` when
  `'serviceWorker' in navigator && document.hidden` rather than `new Notification(...)`.
  Mock `navigator.serviceWorker.ready` to return a fake registration with a `showNotification`
  spy.
- Pre-fix failure mode: the current module only does `new NotificationCtor(...)` which
  Chrome on Android will simply drop when the tab is backgrounded. The new test asserts
  the Service-Worker path is taken; current code has no Service-Worker path so the test
  wouldn't even import.
- E2E: drive Playwright against a real Chromium in mobile emulation mode (`deviceScaleFactor`
  and mobile viewport), grant Notification permission programmatically, background the tab
  via `page.evaluate(() => document.visibilityState)` + a second tab, and assert that a
  Service-Worker `showNotification` call was made (captured by installing a test Service
  Worker that logs to the page). Pre-fix, this would time out or never see a notification
  delivered.

### Bug 3: Banner Enable button no-op on desktop; repeated clicks trigger auto-deny

- File: `tests/notifications-banner.test.ts` (new)
- What it exercises: mount a minimal DOM containing the banner elements from `index.html`
  (or use `happy-dom`). Initialize `createNotifications` with `FakeNotification.permission
= 'default'` and a `requestPermission` mock. Simulate a click on the Enable button.
  Assert `requestPermission` is called. Now simulate the prompt being dismissed
  (`permission` stays `'default'`). Simulate another click. Assert one of:
  (a) the click no-ops with a user-visible message, or (b) the click tracks a counter and
  escalates to a modal after N attempts instead of re-firing `requestPermission()`.
- Pre-fix failure mode: the current `updateBanner` keeps showing the Enable button while
  state is 'default', and each click calls `notif.requestPermission()`. Chrome tracks
  rapid repeated permission prompts and will auto-deny. The test asserts that clicking
  3x without a decision does not call `requestPermission` a 3rd time; pre-fix, it would
  have called it 3x → test fails.
- Also add: after `requestPermission` resolves with `'denied'`, the button should be hidden
  on the next `updateBanner()`. `dashboard.js:2018-2020` already handles this; verify it
  in the test.

### Bug 4: Dashboard-as-recipient messages missed

- Existing fix coverage: `tests/notifications.test.ts:351-395` covers the notification
  module. **But** the end-to-end path is still untested. Add:
- File: `tests/serve-websocket.test.ts` (new — requires starting `Bun.serve` in-process or
  factoring the server out of `serve.ts`)
- What it exercises: start the serve harness, connect a WS client, feed an envelope with
  `to='dashboard'` and `from='research'` through the monitor's onMessage callback, assert
  the client receives a `{type:'message', data: envelope}` frame. Separately, assert that
  `buildPermissionRequestFrame(envelope)` correctly returns null for a plain message so the
  fallback `{type:'message'}` path is taken.
- Pre-fix failure mode: historically, the recipient-check in `notifications.js` said
  `envelope.to === sessionName || envelope.to === 'all'` — so a message addressed to
  'dashboard' was only ever shown on the dashboard client itself (not surfaced as a
  notification for the sender's session). The test would assert that the notification path
  does fire for the sender's session when `to='dashboard'`; pre-fix, `fired.length` would
  be 0.

### Additional broader integration tests to prevent the "cracks"

1. **Two-process MCP conversation** — `tests/e2e/two-sessions.spec.ts`. Spawn two
   `bun src/server.ts` subprocesses with distinct `PARTY_LINE_NAME`s. Open their stdio.
   Send a `tools/call` for `party_line_send` from A targeting B, wait for an MCP
   notification on B's stdout. Asserts the full transport + protocol + delivery loop.
   This is the single highest-ROI test missing — if it exists and passes, a large class
   of regressions becomes impossible.

2. **Ingest → aggregator → WS broadcast** — `tests/e2e/ingest-pipeline.spec.ts`. Boot
   `dashboard/serve.ts` in-process with a temp DB. Open a WS client. POST a hook event to
   `/ingest` with the valid token. Assert a `{type:'hook-event'}` frame lands at the WS,
   and a `{type:'session-update'}` frame follows from the aggregator.

3. **HTTP route matrix** — `tests/serve-routes.test.ts`. Table-driven test of every HTTP
   endpoint: `/api/sessions`, `/api/overrides` (GET/POST/DELETE), `/api/quota`,
   `/api/permission-response` (valid + invalid bodies), `/api/history`, `/api/send` (valid
   - missing fields), `/ingest` (covered), `/api/session`, `/api/events`, `/api/transcript`
     (with + without `after_uuid`), `/api/sparkline`, `/api/self`, `/api/machines`, static
     assets (cache, CSS, JS, notifications.js). For each, assert status code, content type,
     and response shape. Currently every one of these endpoints is deployed untested.

4. **UDP transport loop** — `tests/transport-loop.test.ts`. Create two
   `UdpMulticastTransport` instances on different random ports on the same loopback group.
   Send envelope from A. Assert B receives exactly once. Stop B, restart, send again with
   same envelope id, assert dedup works across restart (or test that dedup state is
   purposely not persisted, depending on design intent). Test `includeSelf` flag.

5. **Client router state machine** — `tests/client-router-logic.test.ts`. Refactor
   `applyRoute` and friends to take `(state, ctx)` where ctx carries `lastSessions`,
   DOM hooks, and navigation. Table-driven: pairs of (initial URL, WS event sequence,
   expected final DOM state). This catches Bug 1 and prevents a large class of future
   routing bugs without standing up a full browser.

6. **Playwright smoke test** — `tests/e2e/playwright-smoke.spec.ts`. Boot serve.ts, drive
   Chromium to the dashboard, verify: (a) homepage loads, (b) `/session/foo` deep link
   works after WS arrival, (c) granting Notification permission via
   `context.grantPermissions(['notifications'])`, (d) sending an event through /ingest
   causes a visible change in the DOM. This one Playwright spec would have caught Bugs 1,
   2, 3, 4 individually.

## Recommendation

Invest in integration and browser-level tests. The unit tests are solid for what they
cover; continuing to add more `FakeNotification`-style unit tests will not catch the
class of bugs that's been shipping. Specifically:

**Tier 1 (do this week):**

1. Add a `test` script to `package.json` (`"test": "bun test"`) so the suite is
   discoverable. Add a `pretest` hook or a CI config that runs `bun run typecheck` +
   `bun test`. Currently the suite is effectively invisible to contributors.
2. Write the two-process MCP conversation test (#1 above). Highest ROI — directly
   validates the product's core promise.
3. Write the ingest pipeline test (#2 above) in-process against `serve.ts`. Medium effort,
   catches a whole class of wiring regressions.
4. Extract the client router state machine (Bug 1 test). Refactor
   `parseUrl`/`applyRoute`/`updateSessions` into a testable module. This is a small
   surgery on `dashboard.js` and gives you a unit-level defense against routing races.

**Tier 2 (next):**

5. Add Playwright and write **one** smoke spec that exercises the full stack including
   notifications with `context.grantPermissions(['notifications'])`. This is the lightest
   viable e2e net. Budget: 1–2 days to set up Playwright + write the spec.
6. Fill in HTTP route matrix test (`tests/serve-routes.test.ts`) — ~12 endpoints, each
   with 2–3 cases. Mostly mechanical.
