# Server / Protocol / DB Audit — 2026-04-20

## Summary

The project is functionally coherent — the protocol round-trips, the observers
don't crash, and the dashboard renders. But the foundation has **accumulated
tech debt in predictable places**: every write is an O(N) fan-out across WS
clients with its own `JSON.stringify`, every INSERT is an unprepared
`db.query(...)` built on the hot path, the aggregator does an extra `SELECT`
per event to rehydrate the row it just wrote, and several "documented"
guarantees (dedup across restart, foreign keys, migrations monotonically
moving forward) aren't actually enforced.

The specific hot-spot that likely explains the "realtime feels inconsistent"
user complaint is probably **aggregator-driven WS fanout**: every hook event
triggers `INSERT → SELECT → JSON.stringify → N×ws.send`, and the aggregator
fires the listener even when the event didn't change any field the UI cares
about. On top of that, hook events, jsonl updates, and the 5s periodic
`sessions` push all race to update the same session card without any
last-writer-wins or version tag — so the UI visibly flickers.

The biggest correctness bugs are:

1. **Deduplicator never drops entries** when messages arrive steadily (it
   only prunes on a 30s timer — and `isDuplicate()` adds the ID before
   checking the window, so the window is meaningless as written).
2. **Migration runner has no transaction** — a partial failure leaves the
   schema in an unbootable state with no rollback and `user_version`
   advanced past the last successful step.
3. **`machines` table is declared, indexed, and queried but never
   written** — `/api/machines` always returns `[]`.
4. **Send-twice is not actually double-sent** — the second send is
   `setTimeout`-scheduled after the first `await` resolves, and if the
   socket errors between the two, the promise from `.send()` has already
   resolved. The caller has no way to know the second hop failed.
5. **`seq` field is per-process, never checked, never transmitted
   usefully** — it's dead weight on every datagram and hidden in the
   types contract that nothing reads.

Below, severity labels are CRITICAL (will cause data loss / visible bugs),
IMPORTANT (silent correctness issue or real perf cost under load), NIT
(code cleanliness / future-proofing).

## Correctness

### CRITICAL — Deduplicator's sliding window is broken

- **Location:** `src/protocol.ts:81-93`
- **Symptom:** A duplicate that arrives _just before_ the 30s prune runs
  is treated as unique forever (correct). But the opposite case — a
  duplicate that arrives from a delayed network buffer after the pruner
  already evicted the first copy — will be **re-delivered**. Worse, the
  dedup window is documented as 60s (`DEDUP_WINDOW_MS`) but the pruner
  runs every 30s in `udp-multicast.ts:62`, and the check `if (ts < cutoff)`
  uses `cutoff = Date.now() - DEDUP_WINDOW_MS`. So an entry is only
  considered "expired" after 60s, but the prune loop runs at 30s.
  Fine — but `isDuplicate()` inserts the new `(id, now)` into the map
  _without_ checking if an existing entry is still inside the window. The
  window check doesn't gate anything; it only gates GC. Result: all
  fresh IDs behave correctly, but there's no bounded-memory guarantee if
  the prune timer misses for any reason (e.g. event loop blocked).
- **Root cause:** `isDuplicate` is a pure set-membership test with a
  side-channel pruner. The window isn't actually a "dedup window" in
  protocol terms — it's a GC heuristic.
- **Fix sketch:** Either (a) make prune synchronous on every `isDuplicate`
  call (cheap, bounded), or (b) cap the map size with an LRU. Also bump
  the pruner interval >= `DEDUP_WINDOW_MS` or drop the separate constants.
  And rename the symbol: it's a "seen set with TTL-based GC", not a
  deduplication window.

### CRITICAL — Sender-restart collision: `seq` resets to 0 but `id` is 16 hex chars of `randomBytes`

- **Location:** `src/protocol.ts:9,12-14`
- **Symptom:** `sequenceCounter` starts at 0 on each process. If a
  session restarts within the 60s dedup window and happens to collide on
  `id` (probability ~2⁻⁶⁴ per message — effectively zero), a receiver
  would silently drop the new message. The real risk is not collision
  but that `seq` gives a false sense of a protocol-level sequence
  number: nothing validates it, nothing uses it to detect dropped
  frames, and it's not per-peer. It's just a process-local counter that
  bloats every envelope.
- **Root cause:** `seq` was specified but never implemented past
  "increment a counter on create."
- **Fix sketch:** Either remove the `seq` field (breaking change, easy)
  or actually use it — per-peer monotonic counters, with receiver-side
  gap detection and "have I seen a newer frame from this peer?"
  filtering. Remove is simpler and honest.

### CRITICAL — Migration runner is non-atomic and asymmetric with schema.sql

- **Location:** `src/storage/db.ts:34-52`
- **Symptom:** `openDb()` runs `schema.sql` _first_ (which includes
  `PRAGMA user_version = 3` at the top of the file), then applies
  migrations. For a brand-new DB this is fine. For an existing DB at
  `user_version = 1`:
  1. `db.exec(schema)` runs. Because `schema.sql` contains
     `PRAGMA user_version = 3`, the user_version is immediately set to
     3 before any migration runs.
  2. `applyMigrations()` reads `user_version = 3` and skips all
     migrations.
     So an upgrade from v1 or v2 **never runs the v2 or v3 migrations**. The
     reason nothing broke is that the migrations are mostly `CREATE TABLE
IF NOT EXISTS` and `ALTER TABLE ADD COLUMN IF NOT EXISTS` duplicating
     work `schema.sql` would do via `CREATE TABLE IF NOT EXISTS`. But the
     v3 migration (`ALTER TABLE events ADD COLUMN source`) is **not idempotent**
     under Bun's sqlite — re-running it throws "duplicate column". Running
     the schema after the migration would be fine; running the schema
     before the migration and then running the migration on a fresh db
     would blow up.
- **Additionally:** `PRAGMA user_version = $next` is outside a
  transaction. If migration N throws halfway through, the PRAGMA isn't
  updated (good), but the partial DDL from that migration may have
  partially committed (DDL in SQLite is transactional only inside
  `BEGIN/COMMIT`), leaving the DB inconsistent with the user_version.
- **Root cause:** `schema.sql` carries `PRAGMA user_version = 3` as a
  declaration, but `db.ts` also manages `user_version` imperatively.
  Two sources of truth.
- **Fix sketch:**
  1. Remove `PRAGMA user_version = 3` from `schema.sql`. Let `db.ts`
     be the sole owner.
  2. Wrap each migration in `db.transaction(() => migration(db))()`.
  3. Decide what "fresh DB" means: either run schema.sql + set
     user_version to the latest, or run schema.sql-for-v0 and apply
     every migration. Currently it's a fuzzy middle.
  4. Add a test that upgrades a v1 DB through v2 → v3 end to end.

### IMPORTANT — `insertEvent` can succeed while aggregator.ingest() fails (or vice versa)

- **Location:** `src/ingest/http.ts:32-36`, `dashboard/serve.ts:310-321`
- **Symptom:** `handleIngest` calls `insertEvent(db, ev)` then
  `opts.onEvent(ev)`, which calls `aggregator.ingest(ev)` (another set
  of INSERTs into `sessions` / `tool_calls` / `subagents`). If the
  aggregator write throws, the event is already in `events` but no
  session row was upserted. On the next page refresh you see a dangling
  event with a session the UI has never heard of. There is no
  transaction wrapping both.
- **Root cause:** Separation between "raw event store" and "derived
  state" with no shared unit of work.
- **Fix sketch:** Wrap both writes in a single `db.transaction(() => { ... })()`
  so either both land or neither does.

### IMPORTANT — Aggregator upserts a fabricated `session_id` when the event carries empty string

- **Location:** `src/aggregator.ts:45`
- **Symptom:** `session_id: ev.session_id || \`${ev.machine_id}:${ev.session_name}\``.
The hook emitter (`hooks/emit.sh:42`) uses `jq -r '.session_id // ""'`so unknown session_id becomes the empty string. The synthetic fallback
produces a stable key per machine+name, but it collides across
different real Claude Code sessions that happen to share a name on the
same machine. Worse, if a later event arrives with the real UUID, the
aggregator writes a second row and the subagent rows recorded under
the synthetic key are orphaned (their`session_id` FK doesn't match).
- **Root cause:** Trying to synthesize a primary key from derived data
  when the real one is missing.
- **Fix sketch:** Skip the upsert when `session_id === ''`, or hold
  events in a small buffer until the first event with a real UUID for
  that session name arrives. Alternatively, add a reconciliation pass
  that rewrites synthetic-key rows when the real UUID shows up.

### IMPORTANT — Observer cannot detect file rotation when replacement produces an identical fingerprint

- **Location:** `src/observers/jsonl.ts:118-144`
- **Symptom:** The fingerprint is the first 64 bytes. Two JSONL files
  that start with the same first line (e.g. a SessionStart record with
  the same `sessionId`, same ISO timestamp to the minute) will collide.
  The observer will then rewind `offsets.set(path, size)` but skip the
  reset callback path that forces a full refetch. The client would be
  locked to an incremental view of the new (smaller) file and miss its
  contents.
- **Root cause:** 64 bytes is not enough to distinguish two session-start
  records that were written within the same second.
- **Fix sketch:** Either (a) include the file's inode/birthtime in the
  fingerprint (`statSync().ino` changes on replacement), or (b) always
  emit `onReset` on any shrink regardless of fingerprint. (b) is
  simpler and matches the current client expectation.

### IMPORTANT — Observer drops entries that straddle a poll boundary

- **Location:** `src/observers/jsonl.ts:155-172`
- **Symptom:** `readSync(fd, buf, 0, length, prev)` reads `size - prev`
  bytes. If Claude Code is mid-write and only part of a JSONL line is
  flushed, the tail slice ends mid-line. `tail.split('\n')` will produce
  a last element that's a partial JSON string; `JSON.parse` throws;
  entry is silently dropped; next poll reads from `size` (which we
  already set), so the completed line never gets re-read.
- **Root cause:** We advance `offsets.set(path, size)` _before_ parsing,
  and our read range is exclusive of the retry zone.
- **Fix sketch:** After parsing, if the final line didn't end in `\n`,
  rewind `offsets` to `prev + <bytes consumed up to last complete line>`.
  Or parse from `prev`, find the last newline, only advance offset to
  that point.

### IMPORTANT — GeminiTranscriptObserver re-reads the whole file on every modification

- **Location:** `src/observers/gemini-transcript.ts:62-104`
- **Symptom:** Gemini's `session-*.json` is a single JSON document (not
  JSONL). Every mtime change triggers `readFileSync` + `JSON.parse` of
  the _entire_ file. For a long session (hundreds of messages, hundreds
  of KB), this is O(n) CPU per poll cycle, and polling is every second.
  If Gemini is actively writing a big file, every write cycle also
  blocks the Bun event loop for the duration of the parse.
- **Root cause:** No alternative — Gemini doesn't write append-only
  JSONL.
- **Fix sketch:** Throttle to at most one parse per N seconds per file,
  or only parse when mtime has been stable for >500ms (i.e. wait for
  the writer to quiesce). Also consider moving parsing to a worker.

### IMPORTANT — Permission-bridge: no timeout on pending requests

- **Location:** `src/permission-bridge.ts:38,41-45`
- **Symptom:** `pending.set(request_id, params)` has no expiration. A
  session that sends a permission-request and then dies (or the
  dashboard responds via a different path, or the envelope is lost
  entirely) leaves the entry in the map forever. With many prompts
  over a long-running session this is an unbounded leak.
- **Root cause:** No TTL, no size bound, no reaper.
- **Fix sketch:** Store `{params, createdAt}` and GC entries older than,
  say, 5 minutes. Also emit a synthetic "deny" MCP notification when an
  entry is GC'd so Claude doesn't hang forever on a lost response.

### IMPORTANT — Presence tracker double-sends on rename

- **Location:** `src/presence.ts:99-105`, `src/server.ts:450`
- **Symptom:** `rename()` deletes the old-name key from the local
  `sessions` map, then sends an announce with the new name. But
  `transport.rename()` was called before `presence.rename()`, so the
  first multicast message goes out under the new name before the
  announce — meaning any peer's heartbeat handler sees traffic from a
  "new" session without the announce metadata. Also, peers still have
  the old name in their `sessions` map until their own
  `SESSION_TIMEOUT_MS` (75s) expires. There's no graceful
  deregister/farewell.
- **Root cause:** Rename was bolted on without a peer-side "bye" event.
- **Fix sketch:** Add an envelope type `bye` (or a `status`-typed message
  with a specific body) that peers treat as a hint to remove the old
  name immediately. Send it before the new announce.

### NIT — `party_line_respond` envelope uses `to=<requester>`, but `response_to` is the callback_id not the original message id

- **Location:** `src/server.ts:379`, `src/protocol.ts:30-46`
- **Symptom:** The envelope field is named `response_to` but holds the
  `callback_id`. The protocol docs describe `callback_id` and
  `response_to` as distinct concepts, but in practice the code uses
  them interchangeably. It works, but the naming is a trap.
- **Fix sketch:** Rename `response_to` → `response_to_callback_id` in
  the TS type and doc, or unify on `callback_id` (request carries it,
  response echoes it; one field, two roles distinguished by `type`).

### NIT — `deserialize` does not check `envelope.seq` presence/type

- **Location:** `src/protocol.ts:54-71`
- **Symptom:** `seq` is declared `number` in the type, but `deserialize`
  validates only id/from/to/type/body. A malicious or buggy sender
  could send `seq: "hello"` and we'd happily propagate it. Low impact
  because `seq` is never read, but inconsistent with the type.
- **Fix sketch:** Either enforce the type in `deserialize` or remove
  `seq` (see CRITICAL above).

### NIT — `session_id` can be empty in the events table

- **Location:** `src/ingest/http.ts:32-33`, schema `events.session_id NOT NULL`
- **Symptom:** The emitter writes `session_id: ""` (jq fallback).
  `NOT NULL` is satisfied by an empty string. Queries that filter by
  `session_id` treat `""` as a real value. The aggregator's
  `getSession('')` returns `null`.
- **Fix sketch:** Reject empty `session_id` in `validateHookEvent`, or
  convert `""` → `null` and make the column nullable, then filter out
  null session events from aggregator sessions-list queries.

## Performance

### IMPORTANT — WS fan-out is O(N) JSON.stringify per event

- **Location:** `dashboard/serve.ts:94-153, 164-172, 316-318`
- **Symptom:** For each emitted frame (`session-update`, `jsonl`,
  `stream-reset`, `message`, `cross-call`, `sessions`, `quota`,
  `hook-event`, `permission-request`), the code does `JSON.stringify`
  **once** and loops `ws.send(json)` — that part is fine. But every
  aggregator tick does **three** stringifies (session-update,
  optionally message, optionally cross-call) and every hook event
  triggers an aggregator write **plus** a hook-event broadcast. With
  5 connected clients, every Bash tool use produces ≈ 6 stringifies and
  15 ws.send calls in a tight loop.
- **Root cause:** No batching or coalescing. Every fine-grained event
  is pushed independently. No microtask-level debounce.
- **Fix sketch:** Add a per-tick coalescer: on the server, accumulate
  outgoing frames in a list during a microtask, then emit a single
  `{type: 'batch', frames: [...]}` at the end of the tick. Client
  unpacks. Also, on the session-update path, suppress emission if
  none of the UI-relevant columns (`state`, `last_seen`, `current_tool`,
  `model`) actually changed.

### IMPORTANT — Every INSERT rebuilds a new prepared statement

- **Location:** `src/storage/queries.ts:49-71, 74-104`, `src/aggregator.ts:57-94`,
  `src/storage/metrics.ts:61-67`
- **Symptom:** `db.query('INSERT ...').run({...})` calls `db.query()`
  **every time** an event arrives. Under Bun's `bun:sqlite`,
  `db.query()` prepares+caches internally, so the second call is
  cheap — but each call still costs a hash lookup, and more
  importantly: there's no `BEGIN; ... COMMIT;` grouping. Every
  `insertEvent` + `upsertSession` + `INSERT INTO tool_calls` is its
  own WAL transaction. A busy session can easily emit 50 tool calls
  per minute → 150+ separate commits.
- **Root cause:** No batching, no cached `Statement` handles.
- **Fix sketch:**
  1. In a module-scoped `prepareStatements(db)` function, prepare every
     statement once and export bound runners.
  2. Wrap `handleIngest`'s `insertEvent + aggregator.ingest` in
     `db.transaction(() => { ... })()`.
  3. For `rollupDailyMetrics`, wrap the whole loop in a transaction —
     currently every day × session does its own commit.

### IMPORTANT — Aggregator does a SELECT after every write just to emit the listener

- **Location:** `src/aggregator.ts:97-100`
- **Symptom:** After each `ingest`, we re-`getSession(ev.session_id)`
  and emit. That's an extra `SELECT * FROM sessions` per event — purely
  to hydrate a row we just wrote. Doubles the DB work.
- **Root cause:** The upsert helper doesn't return the row.
- **Fix sketch:** Have `upsertSession` return the resulting row (using
  `INSERT ... RETURNING *` in bun:sqlite). Or assemble the outgoing
  `session-update` object from the input + prior cached state without
  a round trip.

### IMPORTANT — `hourlyToolCalls` runs on every sparkline fetch without using `metrics_daily`

- **Location:** `src/storage/metrics.ts:88-107`, called from
  `dashboard/serve.ts:369-372`
- **Symptom:** The comment says "sparkline derived from daily metrics
  rollup." But `hourlyToolCalls` SELECTs from the raw `events` table
  with `hook_event = 'PostToolUse' AND ts >= cutoff`. For a busy machine
  with the full 30-day retention, that's a scan of potentially 100K+
  rows for every sparkline render on every session card, on every
  5-second `sessions` push. There's an index on
  `(session_id, ts)` but the query filters on `session_id OR session_name`,
  which **forces a table scan** (index can't be used with OR on two
  columns).
- **Root cause:** Dual-key lookup made the index useless. The metrics
  rollup is orthogonal — it's daily, not hourly.
- **Fix sketch:**
  1. Resolve `sessionKey` to a UUID in the server layer once, then query
     by `session_id` only.
  2. Add an `hour` bucket to a new table, rolled up incrementally on
     ingest.
  3. Cache the result per session for at least the poll interval —
     nothing new arrives between polls, so don't recompute.

### IMPORTANT — `/api/transcript` rereads the full JSONL file on every fetch

- **Location:** `dashboard/serve.ts:345-366`, `src/transcript.ts:384-434`
- **Symptom:** Every call to `buildTranscript` runs `readFileSync` on
  the entire session JSONL. For a long session (>10MB), that's a full
  disk read + parse every time the client polls for incremental
  updates. `filterAfterUuid` slices at the end — after all the work.
- **Root cause:** The incremental mechanism is applied at the wrong
  layer. We do O(n) work to return O(Δ) entries.
- **Fix sketch:** Teach `buildTranscript` to accept a byte offset or
  a "last line" cursor and parse only new lines. The `JsonlObserver`
  already tracks offsets — share the same map.

### IMPORTANT — `monitor.getHistory({limit: 500})` called on every `/api/transcript`

- **Location:** `dashboard/serve.ts:355`
- **Symptom:** Builds a 500-item array, filters heartbeats, slices the
  last N. Fine for small history, but done inside a hot path (every
  transcript fetch). Cost is bounded but non-zero.
- **Fix sketch:** Keep a separate "non-heartbeat history" ring buffer
  in `monitor.ts` so filtering is O(1).

### NIT — Periodic `sessions` push every 5s regardless of change

- **Location:** `dashboard/serve.ts:156-162`
- **Symptom:** Broadcasts the full session list every 5s even when
  nothing changed. With 5 clients and 10 sessions, that's 50 ws
  sends / 5s = 10/s of redundant traffic.
- **Fix sketch:** Hash the session list and skip the push when the hash
  is unchanged. Or replace the periodic push with a presence-change
  event from the PresenceTracker.

### NIT — Introspection cache races

- **Location:** `src/introspect.ts:189-205`
- **Symptom:** `lookupAttempted = true` is set before the lookup runs.
  A concurrent second call (shouldn't happen, but defensively) would
  skip the lookup and see `cachedJsonlPath = null` → return null forever
  even though the lookup succeeded.
- **Fix sketch:** Set `lookupAttempted = true` after both lookups
  complete, or use a lock.

## Realtime Consistency

### IMPORTANT — Multiple update channels race to refresh a session card

- **Location:** `dashboard/serve.ts:94-97, 102-113, 123-153, 156-162, 316-318`
- **Symptom:** A single Bash tool use produces, in rough order:
  1. `hook-event` (PostToolUse)
  2. `session-update` (from aggregator emitting after the insert)
  3. `jsonl` (from the transcript observer ~500ms later)
  4. `sessions` (from the 5s periodic push)

  There's no sequence/version number, so the client renders them in
  arrival order. If the jsonl-derived state reaches the client before
  the session-update (possible under load), the UI first shows the
  JSONL-derived "working on X" state, then the aggregator's state,
  then the 5s periodic push could revert an optimistic local change.
  This is the source of the "flickery" feel.

- **Root cause:** No single source of truth on the server; each
  observer pushes its own view of reality.
- **Fix sketch:** Give every session a monotonic `revision` field in
  the DB (incremented on every update). Every outbound `session-update`
  or `sessions` frame carries the revision. Client drops older
  revisions. A `sessions` push that matches the client's last-seen
  revisions per session is a no-op.

### IMPORTANT — Aggregator is not idempotent for `tool_calls`

- **Location:** `src/aggregator.ts:83-94`
- **Symptom:** Every PostToolUse event does an `INSERT INTO
tool_calls`. There's no UNIQUE constraint, no `ON CONFLICT` clause,
  no idempotency key. If the same hook fires twice (retry, replay,
  upstream duplicate delivery), we get two rows. Metrics and
  sparklines double-count.
- **Root cause:** Hook events are identified only by
  `(machine_id, session_id, hook_event, ts)` implicitly — nothing
  persists that.
- **Fix sketch:** Add an `event_id TEXT UNIQUE` column on `tool_calls`
  and derive it from the PostToolUse payload (Claude Code hooks carry
  a `tool_use_id` in `payload.tool_use_id`, which is the canonical
  idempotency key). Use `ON CONFLICT DO NOTHING`.

### IMPORTANT — Out-of-order events flip session state incorrectly

- **Location:** `src/aggregator.ts:29-36, 44-53`
- **Symptom:** `upsertSession` uses `COALESCE(excluded.state,
sessions.state)` — so state is updated every time an event arrives
  that has a non-null state. But `WORKING_EVENTS` fires `'working'` on
  PostToolUse too (not just PreToolUse). If a `PostToolUse`
  arrives _after_ a `Stop` event (network delay, remote host skew),
  the session is incorrectly re-marked `'working'` even though it's
  actually idle.
- **Root cause:** Event order is assumed to match wall clock. `ts` is
  set by the emitter; remote hosts can have skew; we never check.
- **Fix sketch:** Only update `state` if `ev.ts > sessions.last_seen`.
  Reject late events for state transitions. Keep them in `events` for
  history.

### IMPORTANT — Presence self-record says "just seen right now"

- **Location:** `src/presence.ts:79-89`
- **Symptom:** `listSessions()` always injects `lastSeen: Date.now()`
  for self, regardless of whether the transport is actually running.
  If the transport fails silently, the dashboard still shows itself
  as online.
- **Fix sketch:** Track the last successful `sendHeartbeat()`
  completion timestamp and use that.

### NIT — Dedup set shared across session renames

- **Location:** `src/transport/udp-multicast.ts:20,88-90`
- **Symptom:** When a session renames, the Deduplicator is not
  cleared. Messages originating from the _old_ name (including any
  that were in-flight) continue to be filtered through the same map.
  Fine — IDs are globally unique random — but worth documenting.

## Protocol

### IMPORTANT — `permission-request` / `permission-response` dual-pathed inconsistently

- **Location:** `src/server.ts:185-193`, `src/permission-bridge.ts:41-45`
- **Symptom:** On `handleInbound`, `permission-request` envelopes are
  dropped early ("for the dashboard, not other sessions"), and
  `permission-response` is only accepted if `envelope.to === sessionName`.
  But permission-request is _sent_ from the session to `'dashboard'`
  (hard-coded target). If the dashboard session happens to be named
  something else (`--name mission-control` etc.), the dashboard never
  sees the permission prompt.
- **Root cause:** The target name `'dashboard'` is baked into
  `permission-bridge.ts:44`.
- **Fix sketch:** Either broadcast permission-request to `'all'` and
  let only the dashboard-role listener consume it, or discover the
  dashboard's name via presence and route to it by role tag. A
  `SessionMetadata.role = 'dashboard'` field would make this clean.

### IMPORTANT — `to` supports CSV targeting but only in some code paths

- **Location:** `src/server.ts:196-202`, `src/transcript.ts:417`,
  but **not** `dashboard/serve.ts:137` (the cross-call filter uses
  `envelope.to !== 'all'`)
- **Symptom:** If a user sends to `"alice,bob"`, the cross-call
  visualization records one edge per message, from → `"alice,bob"`,
  rather than from → alice, from → bob.
- **Root cause:** CSV targeting was bolted on at the delivery layer
  without updating the observability layer.
- **Fix sketch:** Parse `to` at the monitor level and fan out
  cross-call frames per target.

### NIT — `TeammateIdle`, `PostCompact`, `TaskCreated`, `TaskCompleted` declared in HOOK_EVENT_NAMES but unused

- **Location:** `src/events.ts:11-16`
- **Symptom:** Aggregator doesn't handle these. `hooks/install.sh:28-29`
  installs `TaskCreated`/`TaskCompleted` hooks. When they fire, they
  pass validation, get inserted into `events`, but produce no visible
  state change. History view shows them; everything else ignores them.
- **Fix sketch:** Either add aggregator branches (map `TaskCreated` →
  subagents table, etc.) or trim the allowlist. Document the gap.

### NIT — `envelope.seq` is in the wire format but has no semantic value

- See CRITICAL above. It's also in the type — honest candidate for
  removal.

### NIT — `permission-response` handler drops silently when `!pending.has(id)`

- **Location:** `src/permission-bridge.ts:57-60`
- **Symptom:** A late or duplicate permission-response with a
  request_id we've already resolved is dropped silently. Fine for
  security, but this is the same class of issue as the dedup window:
  with a TTL on `pending` (see IMPORTANT above), we could distinguish
  "never seen" from "already resolved."

## Storage

### IMPORTANT — `machines` table is declared and queried but never written

- **Location:** `src/storage/schema.sql:7-12`, `dashboard/serve.ts:380-388`,
  `src/ingest/http.ts` (missing)
- **Symptom:** `/api/machines` always returns `[]`. The "Machines" tab
  in the UI can render only whatever data the client has (zero).
  Foreign-key intentions in `schema.sql:15` reference this table.
- **Root cause:** The machines upsert was planned but never written.
- **Fix sketch:** Add `upsertMachine(db, machine_id, hostname,
last_seen)` to `queries.ts` and call it from `handleIngest` before
  `insertEvent`. The hostname can come from the session's first
  `SessionStart` payload or a dedicated machines-register endpoint.

### IMPORTANT — Every table lacks FKs, so retention is incomplete

- **Location:** `src/storage/schema.sql:17-85`, `src/storage/retention.ts`
- **Symptom:** Pruning deletes from `events` only. `tool_calls`,
  `subagents`, and `sessions` keep growing forever. After several
  months the `tool_calls` table is the biggest on disk and the
  aggregator does table scans that take seconds.
- **Root cause:** FKs were deliberately omitted (comment in schema
  explains why — lazy upsert ordering). But retention has to walk
  every table manually; it currently doesn't.
- **Fix sketch:** Extend `pruneOldEvents` to `pruneOldData`:
  - `DELETE FROM events WHERE ts < cutoff`
  - `DELETE FROM tool_calls WHERE started_at < cutoff`
  - `DELETE FROM subagents WHERE started_at < cutoff AND status = 'completed'`
  - `DELETE FROM sessions WHERE last_seen < cutoff AND state IN ('ended', 'idle')`
    Wrap in a single transaction. Log row counts per table.

### IMPORTANT — No WAL autocheckpoint / journal size management

- **Location:** `src/storage/schema.sql:1`, `src/storage/db.ts`
- **Symptom:** `journal_mode = WAL` is set but `wal_autocheckpoint`
  defaults to 1000 pages and nothing runs `PRAGMA wal_checkpoint(PASSIVE)`
  explicitly. Over time the `-wal` file can grow to hundreds of MB if
  long-running transactions ever stall checkpointing.
- **Fix sketch:** After the retention pass, run
  `PRAGMA wal_checkpoint(TRUNCATE)` and log the result.

### IMPORTANT — No `ANALYZE` ever runs

- **Location:** `src/storage/db.ts`
- **Symptom:** SQLite plans queries based on the last `ANALYZE`. A DB
  that grew organically without `ANALYZE` may pick a bad plan for
  queries like `hourlyToolCalls` or the aggregator's JOINs.
- **Fix sketch:** `PRAGMA optimize;` on every startup after migrations.
  Nearly free; re-plans if stats are stale.

### IMPORTANT — `recentEvents` SELECT is unindexed when filtering by session_name

- **Location:** `src/storage/queries.ts:131-146`, schema
  `idx_events_session_ts ON events(session_id, ts)`
- **Symptom:** The OR clause `WHERE session_id = $key OR session_name = $key`
  disables the `(session_id, ts)` index. On a table with 100K+ rows,
  this is a full scan every time the History tab filters by session.
- **Fix sketch:**
  - Add `idx_events_name_ts ON events(session_name, ts)`.
  - Rewrite the query as `UNION ALL` so each arm can use its own
    index; dedupe by `id` if needed.
  - Better: resolve `session_name` → `session_id` once in the caller
    and query by `session_id` only.

### NIT — Schema mixes `TEXT` ISO timestamps with no ordering guarantees

- **Location:** schema.sql throughout
- **Symptom:** `ts` is `TEXT`; comparisons like `ts >= cutoff` rely on
  ISO 8601 lexicographic ordering. Works fine until someone writes a
  timestamp without the trailing `Z` or with an explicit offset. BSD
  `date` in the remote emitter (`hooks/remote/emit.sh:34`) emits
  to-the-second precision when `%N` is unavailable — that's a
  different string length but still lexicographically correct. OK for
  now; document that all `ts` values MUST be Zulu ISO 8601.
- **Fix sketch:** Store as INTEGER (unix millis) with a TEXT view if
  needed. Bigger lift; leave for later.

### NIT — `context_tokens` / `message_count` are INTEGER but aggregator writes strings via JSON

- **Location:** `src/aggregator.ts:44-53`
- **Symptom:** None of these fields are written by the aggregator —
  they're only populated by the MCP server's presence heartbeat path
  (via `SessionMetadata.status`). But `upsertSession`'s `COALESCE`
  ensures they stay `null` if never written. Fine — but the UI shows
  "null ctx" rather than "unknown" or "— ".
- **Fix sketch:** A hook-event path that extracts token counts from
  PostToolUse.tool_response when available. Or accept that these are
  MCP-sourced only and gate the UI column on source = 'claude-code'
  with MCP active.

## Security

### IMPORTANT — `handleIngest` returns the storage error message in plaintext to the client

- **Location:** `src/ingest/http.ts:34-37`
- **Symptom:** `return new Response(\`Storage error: ${(err as Error).message}\`, ...)`.
  An unauthenticated-but-token-holding client (e.g. a misbehaving
  remote hook that the user installed) can probe schema details by
  sending malformed payloads — the DB error is echoed back. Not a
  critical leak given the Twingate boundary, but unnecessary.
- **Fix sketch:** Log the real error server-side, return a generic
  `500 Internal Server Error`. Attach a correlation ID if you want
  debuggability.

### IMPORTANT — `/api/send` and `/api/permission-response` have no auth

- **Location:** `dashboard/serve.ts:254-285, 294-307`
- **Symptom:** The entire REST surface is unauthenticated. `/api/send`
  can be called by anything that can reach the port — publishes
  arbitrary messages as the dashboard. `/api/permission-response`
  can approve tool calls on any session. Behind Twingate this is
  fine; if the user ever opens the port accidentally (e.g. tries to
  test with `0.0.0.0`), it's a hole.
- **Fix sketch:** Require the ingest token on state-changing REST
  endpoints, or scope them to `localhost`/loopback only. Reuse
  `verifyToken` from `ingest/auth.ts`. For the browser session,
  bootstrap via a cookie set by a `GET /api/bootstrap` that bakes a
  one-time token into the HTML.

### IMPORTANT — WS channel accepts send/respond commands with no auth

- **Location:** `dashboard/serve.ts:434-462`
- **Symptom:** Same issue: any WS client can invoke `send`, `respond`,
  `session-viewed`. Same mitigation.

### IMPORTANT — Unbounded payload size on `/ingest`

- **Location:** `src/ingest/http.ts:20-25`
- **Symptom:** `req.json()` reads the full body with no size cap. A
  misbehaving hook could POST a 100MB payload and OOM the dashboard.
  Bun does have some server-level defaults, but nothing sets them
  explicitly here.
- **Fix sketch:** Check `Content-Length` header; reject >1MB.

### IMPORTANT — `indexHtml` et al. loaded at startup and cached

- **Location:** `dashboard/serve.ts:176-180`
- **Symptom:** If the dashboard is running when assets change (e.g.
  dev workflow editing `dashboard.js`), the server serves stale
  bytes. Not a security issue, but it masks hot-reload-style
  development.
- **Fix sketch:** Optional `--dev` flag that re-reads on each request.

### NIT — `/vendor/` path allows traversal only by regex

- **Location:** `dashboard/serve.ts:391-401`
- **Symptom:** `/^[a-zA-Z0-9._-]+$/` rejects `..` — good. But a file
  named e.g. `.env` in `dashboard/vendor/` would match the regex and
  be served. Currently harmless (no secrets are placed there), but
  a cleaner pattern is an explicit allowlist of file names.

### NIT — Ingest token is 64 hex chars, readable by any local process

- **Location:** `src/ingest/auth.ts:18`
- **Symptom:** `chmodSync(path, 0o600)` — good. But the token is
  also readable by `bash` hook scripts running as the same user, which
  is the whole point. Accept this is the threat model.

## Architectural Recommendations

1. **Collapse the "three flavors of push" into one.** Currently the
   server exposes `session-update` (aggregator), `sessions` (periodic
   presence snapshot), and `jsonl` (transcript delta). All three
   update the same UI state. Introduce a single `session-delta`
   stream with a monotonic revision; the client maintains a
   revision-per-session map and ignores anything stale. `sessions`
   becomes a handshake-only frame.

2. **Unify the writer side of the DB behind a service object.** Today
   `queries.ts`, `aggregator.ts`, `metrics.ts`, and `retention.ts`
   each directly call `db.query(...)`. A `Store` class that owns the
   Database and exposes typed methods (`ingestEvent`, `tallyDay`,
   `prune`, `snapshot`) is easier to reason about, easier to wrap in
   transactions, and easier to instrument (timings, row counts).

3. **Separate the transport layer's "known-IDs" cache from the
   protocol layer's dedup window.** Today `Deduplicator` lives in
   `protocol.ts` but is only used by `udp-multicast.ts`. A transport
   that isn't broadcast-reliable (Discord thread, email) shouldn't
   need dedup at all. Move dedup to be a transport concern.

4. **Make "session" a first-class DB concept with a generated UUID.**
   The events table's `session_id` is whatever the emitter decides.
   The aggregator silently fabricates keys from (machine_id, name).
   This produces the dangling-row problem (IMPORTANT above). Having
   the DB own a `session_uid` that the aggregator maps to on ingest
   (with fuzzy matching: machine+name+time) cleans up the identity
   story.

5. **Introduce a per-event idempotency key.** Today we dedupe at the
   UDP transport layer, but hook events have no dedup. A `hook_event_id`
   carried on the wire (SHA1 of the raw payload, or Claude Code's own
   tool_use_id for PostToolUse) plus a UNIQUE index would make ingest
   idempotent and enable safe retries from remote hosts.

6. **Stop doing IO on the aggregator's hot path.** Currently the
   aggregator's `ingest` produces a session-update that includes a
   full re-SELECT. The aggregator should return the shape it already
   holds in memory (or build it from the input). Likewise, sparklines
   should read from a pre-computed table, not scan events.

7. **Unit-test the protocol + migrations.** There are storage tests
   (`tests/storage.test.ts`), but no tests exercise:
   - a v1 → v3 migration upgrade
   - dedup with pruning under bursty sends
   - out-of-order event folding (PostToolUse before PreToolUse)
   - observer recovery from a truncated JSONL line at a poll
     boundary
   - permission-bridge timeout/GC behavior (once added).

8. **Consider removing the `seq` field.** It's dead wire protocol
   weight. Either use it (per-peer sequencing, gap detection, dropped
   frame logging) or drop it. Adding cost without value erodes the
   "clean foundation" goal.

9. **Give the dashboard a role tag in presence metadata.** Hard-coded
   `'dashboard'` in `permission-bridge.ts:44` breaks as soon as the
   user renames. A `role: 'dashboard'` (or `role: 'mission-control'`)
   flag in `SessionMetadata` that senders can look up via
   presence.listSessions() makes permission routing name-agnostic.

10. **Write the `machines` table, or drop it.** `/api/machines` and
    the Machines tab currently surface empty data. Either make the
    table useful (per-host sessions count, last-seen heartbeat
    timestamps, a `status` derived from the most recent event) or
    remove the tab and the table.
